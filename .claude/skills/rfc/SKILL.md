# RFC — persona-pool review of a wiki page

You are working in `~/projects/brain`. The `rfc` skill produces
**multi-perspective review** of a forward PRD or a freshly-written
ADR by simulating reactions from the relevant personas. It exists
because:

- A PRD authored by the PM agent reflects one perspective.
- A real Shape Up review would have backend, platform,
  architect, and PM personas all weighing in before the bet got
  made.
- The brain's persona pool (`.claude/personas/team/` +
  `.claude/personas/users/` + `.claude/personas/agents/`) makes that
  multi-perspective pass *deterministic* — for the same page, the
  same pool reacts the same way.

## When to run

- **Standalone** — `/rfc <wiki-relative-path>` after a page lands.
  Useful for stress-testing an existing PRD/ADR or an org-level
  page that crosses repos.
- **As `/shape --rfc`** — opt-in middle phase between PM (PRD) and
  Tech Lead (ADR). The PM phase produces the PRD; `--rfc` runs an
  RFC pass; Tech Lead reads PRD + RFC reactions before authoring
  the ADR.

The default `/shape` workflow is *without* RFC — RFC adds latency.
Opt in when the work is contentious, novel, or crosses repos.

## Persona-pool inference

The agent picks the pool from the target page's frontmatter, in
this order:

| Frontmatter signal               | Adds to pool                                                      |
|----------------------------------|-------------------------------------------------------------------|
| `team: <name>`                   | Every team persona whose own `team:` matches.                     |
| `repos: [<r1>, <r2>, ...]`       | Team personas whose role spans those repos (judgement call — e.g. a platform-eng persona spans every repo). |
| `affects: [<r1>, <r2>, ...]`     | Same heuristic, broader: this page changes those repos, even if it's not *about* them. |
| `## Affected personas`           | Every named user persona under that section.                      |
| Default (always present)         | The three agent personas — PM agent, Tech Lead agent, Developer agent — unless the page itself was authored by one of them (skip the author). |

If the inferred pool is empty (very small / brain-meta page), fall
back to: PM agent + Tech Lead agent + the brain's "audience"
heuristic (the page's `kind:` — `decision` → architect personas;
`initiative` → PM + EM personas).

Cap the pool at **6 personas** to keep RFC runs bounded. If more
than 6 are inferred, prefer named personas over generic role
personas, and flag the omission in the RFC section.

## Protocol

### 1. Load the target page

Read the page in full. Note its `kind:`, `team:`, `repos:`,
`affects:`, and `## Affected personas` sections.

### 2. Infer the pool (per the table above)

Print the pool to the operator before running the reactions, so
the routing is auditable:

```
RFC pool for wiki/<repo>/prds/<slug>.md:
  - <name> (senior backend, <team>)
  - <name> (platform engineer, Platform)
  - <name> (principal architect, Engineering)
  - <name> (PM, <team>)
  - <name> (<role>, external user persona)
  - Tech Lead agent
```

### 3. For each persona, generate a one-paragraph reaction

Adopt the persona's voice (read their persona file under
`.claude/personas/<level>/<slug>.md`). Each reaction is a single
paragraph (3–6 sentences), structured as:

- **Opening reaction** — what they think, in one sentence.
- **Concerns** — what would worry them about this approach.
- **Counter-proposal or amplification** — what they'd want
  changed or what they'd want preserved.

Do not output bullet lists; prose only. The voice is what
distinguishes the perspectives.

### 4. Append the RFC section

Add to the target page, just before `## Build notes` (if present)
or at the bottom:

```markdown
## RFC

*Generated by `/rfc` on YYYY-MM-DD. Pool inferred from
frontmatter; see the skill for the inference rule.*

**<Name> — Senior Backend Engineer.** *(reaction paragraph)*

**<Name> — Platform Engineer.** *(reaction paragraph)*

**<Name> — Principal Architect.** *(reaction paragraph)*

**<Name> — Product Manager.** *(reaction)*

**<Name> — <role> (user persona).** *(reaction)*

**Tech Lead agent.** *(reaction — usually the most weight, since
this is the agent that will own the resulting ADR)*
```

### 5. Update the page's frontmatter and log

- Bump `updated:` to today (UTC).
- If the page is `confidence: low`, leave it — RFC pass alone
  doesn't graduate confidence. The Tech Lead reading the RFC
  reactions does that during the next `/shape` phase.
- Append to `log/log.md`:

  ```
  YYYY-MM-DD rfc — wiki/<path>: <N> persona reactions
  ```

## Hand-off mechanics

When invoked as `/shape --rfc`, the `shape` skill runs:

1. **Phase 1 (PM agent)** — produce the PRD as usual.
2. **Phase 1.5 (RFC)** — `rfc` skill runs against the new PRD;
   appends the `## RFC` section.
3. **Phase 2 (Tech Lead agent)** — reads PRD *plus* RFC, then
   authors the ADR. The ADR's `## Alternatives` should reference
   the RFC reactions where relevant ("the backend persona's concern
   about N+1 queries on the list view maps to Option B above").

Standalone `/rfc <page>` skips the workflow shaping; it just
appends reactions to the existing page.

## Done check

- [ ] Pool is announced to the operator before reactions are
      generated.
- [ ] Pool size is ≤ 6 personas.
- [ ] Each persona's reaction is a single paragraph in their voice
      (no bullet lists).
- [ ] `## RFC` section is appended at the right place (before
      `## Build notes`).
- [ ] `updated:` is bumped to today (UTC).
- [ ] `log/log.md` has an `rfc` line.

## What `/rfc` is *not*

- Not a substitute for human review on contentious decisions —
  RFC reactions are simulated, not real teammates'.
- Not a confidence-bumping mechanism — only the Tech Lead agent's
  ADR (or human approval) graduates confidence.
- Not for `kind: reference` pages. Reference describes
  state-of-the-world; reactions don't add value. RFC targets
  `initiative` / `decision` / `meta` (when meta makes a
  cross-cutting argument).
