Records the decision that shaped the wiki home page contract: what
the home page is, who maintains it, and how that maintenance is
enforced.

## Context

The brain's UI serves `wiki/index.md` at the site root, and agents
read the home page as their first priming context on nearly every
operation. Historically the file was a curated catalog — a table of
contents over the corpus. Three forces pushed for more: the mission
statement that agents use the brain as primary working memory needs
a current-state primer, not just navigation; the brain should apply
its own Now / Perceived / Target discipline to itself; and the UI
substrate swap (see [successor-ssg-for-ui.md](successor-ssg-for-ui.md))
had just settled, making content shape the next open question.

The decision repurposes `wiki/index.md` itself as an
agent-maintained dashboard. No new file is created; the old catalog
content folds in as a final wayfinding section. Seven dashboard
sections render in a fixed order — What changed, Open initiatives,
Recent decisions, Drift surface, Insights now, Brain trajectory,
Curated picks — followed by Where to find things. The ordering
rationale is time-sensitive first, risk second, in-flight work
third, synthesis fourth, editorial fifth, wayfinding last.

The section anchors are a contract, not a convention. Headings are
the anchors (generated by the same slugger the substrate pins), so
renaming a heading is a breaking change. Machine consumers read a
supplementary sections field in the generated pages index rather
than parsing markdown. Every section heading renders even when
empty: the empty state is a single italic placeholder line plus a
machine-readable HTML comment marker, so tooling can distinguish
"maintained but currently empty" from "abandoned".

Maintenance is owned by the skill that produces each section's
content — each relevant skill gains a done-check item to update its
owned section — rather than by a central writer. Enforcement is a
hook pair plus a CI guard. A PostToolUse hook watches edits under
`wiki/` (excluding the views and archive folders and the home file
itself) and sets a staleness sentinel; a Stop hook reads the
sentinel and blocks the agent from finishing a turn that edited
wiki content without touching the home page. Editing the home file
clears the sentinel. The CI guard is the load-bearing enforcement —
any change set that modifies wiki content must also modify the home
page — while the local hook is ergonomics for the common
accidental-forgetting case. A hook script failure halts the turn
loudly; silent passes are forbidden. Both halves live in
`tools/home-staleness.sh` (see the script header for the wiring).

For human viewports the page uses progressive disclosure: two
sections expanded by default (What changed and Drift surface), the
rest collapsed; agents parse everything expanded. Finally, the
decision commits to one canonical primer — audience-specific home
pages are future rendering tweaks on the same file, never sibling
primers.

The decision carries an exit ramp: it re-opens if the section count
grows past nine, if one section routinely dominates the page, or if
the hook-plus-CI enforcement misses a meaningful share of bypass
cases. Re-opening means a new recorded decision, not silent drift.

## Alternatives

- **The wiki index as dashboard, catalog folded in** *(chosen)* —
  a single file at the site root, native routing with no redirect
  config, and one maintenance responsibility per ingest. Cost: the
  standalone curated-catalog convention dies.
- **New home file plus a root redirect, index stays a catalog** —
  cleanest conceptual separation, but it doubles the maintenance
  surface (every ingest updates two navigation-shaped pages),
  requires substrate config for the redirect, and leaves agents
  walking `wiki/` unsure where to land. Rejected.
- **Home page under the brain's own shelf** — symmetrical with how
  repos get state pages, but it doesn't render at the root without
  a redirect and conflates the brain's self-tracking with the UI's
  entry point. Rejected.
- **Multiple home pages, one per audience** — bakes audience modes
  into v1, multiplies maintenance across every skill, and the Stop
  hook doesn't generalise cheaply to N enforcement targets.
  Rejected; one primer, audience modes later if ever.
- **Status quo (curated catalog only)** — consistent but leaves the
  priming mission unmet. Rejected.

## Consequences

- *Closes:* the "two files compete for the root" question; the "is
  the home stale?" question (CI makes it machine-checkable, the
  hook makes forgetting hard); the risk of anchor drift for machine
  consumers (the anchor list is contract); the risk of audience
  modes leaking into v1.
- *Opens:* a clean extension point in the generated pages index
  that future per-audience or per-team rendering can consume; a
  reusable hook pattern — PostToolUse plus Stop with a sentinel and
  a CI guard — for any future "this file must be touched when that
  file is touched" rule; the home page as a first-class agent
  surface that lowers the cost of corpus priming.
- *Costs:* the catalog becomes one section near the bottom instead
  of a top-level concept; several skills gain a coordinated
  done-check item; one new tool script plus a small validator
  extension enters the maintenance footprint; the editorial
  rotation policy for Curated picks is deliberately left to the
  build phase rather than fixed here.

## Build notes

The sentinel lives in the gitignored views folder and records the
list of wiki paths edited since the home page was last touched, so
the Stop hook's blocking message can name exactly what went
unreflected. The hook reads the edited path from the harness's tool
input on stdin and normalises absolute paths against the repo root.

The section contract proved extensible in practice: a later
amendment added a section surfacing agent-authored drafts awaiting
a specific human role's input. The lesson generalises — extensions
to the anchor list are legitimate, but only as explicit,
operator-approved contract changes recorded against this decision,
never as ad-hoc headings.
