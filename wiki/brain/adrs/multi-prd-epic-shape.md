# Epics are single-page umbrella artifacts with `parent_epic:` linkage, validator-enforced no-umbrella-ADR, epic-aware briefs, and a quiet-on-day-one promotion heuristic

Brain-meta decision: initiatives too large for a single PRD/ADR pair — the "migrate framework A to framework B" or "adopt OAuth2 across all services" scale — get a first-class umbrella artifact, `kind: epic`, living as a single page at `wiki/<scope>/epics/<slug>.md`. The umbrella is PRD-shaped (Objective, Background, Affected personas, Scope, No-gos, Children, Success metrics) but pairs with **no ADR** — the validator enforces this, because the umbrella coordinates children rather than making engineering bets. Children (regular PRDs and ADRs) declare membership through a `parent_epic:` frontmatter field, a single string that must resolve to an existing epic page; the validator fails on dangling references. Reverse edges (`child_prds`, `child_adrs` split by kind) are computed by the views tooling through the same code path as the existing `affects:`/`affected_by:` loop, and a by-epic view is emitted alongside the other generated indices.

The [/shape skill](../../../.claude/skills/shape/SKILL.md) gains an `--epic` mode that runs Phase 1 (PM authoring plus optional RFC pass) against the umbrella narrative and explicitly skips the ADR phase. Regular `/shape` gains two pre-flight behaviours: an epic-detection prompt (when epics exist in the target scope, one line asks whether the new pitch belongs to one; default is decline) and a soft-promotion suggestion that proposes `--epic` only when *all* size signals fire together — big appetite AND multiple subsystems AND at least three distinct personas. The all-or-nothing gate is the quiet-on-day-one heuristic: false positives are expensive, false negatives are recoverable, so the asymmetry favours quiet. Children carry an aside linking back to the parent epic, captured as a spawn-time snapshot of the umbrella's objective (no auto-sync on parent edits in v1). The epic page's Children section is hand-maintained by the agent at child-spawn time and on lifecycle advances. The [/zoom-out skill](../../../.claude/skills/zoom-out/SKILL.md) becomes epic-aware: a brief on a child inserts an umbrella-narrative excerpt from the parent, and a brief targeted at an epic itself uses a four-section shape (umbrella narrative, children with statuses, cross-cutting edges, product fit), all within the existing synthesis prompt budget. `/continue` surfaces the parent epic alongside a child's state. Closing the epic is manual in v1: the operator bumps the status when the umbrella completes; the agent may suggest the bump when it notices all children have shipped but never acts alone.

## Context

The PRD behind this decision asked for an umbrella shape for multi-PRD initiatives, and its review pass added the load-bearing constraints absorbed above: the kind distinction should be schema-enforced with validator teeth rather than convention alone; view aggregation must stay linear in corpus size and reuse the existing reverse-edge code path rather than introduce a parallel one; the epic-targeted brief shape — the most expensive piece — had to be specced concretely before committing, with a documented fallback to frontmatter-only awareness if it didn't fit a single prompt template; the promotion heuristic had to stay quiet by default (a prior build's iteration cycle taught that false-positive suggestions are costly); and children needed an inline status line on the umbrella plus a visible link back to the parent for non-technical readers.

A pre-decision investigation of the existing schema tooling confirmed the extension is small: adding the kind is one line, the reverse-edge loop takes one new key, and the no-umbrella-ADR rule is a single path-existence check. The audit of the brief shape found the technical half is pure structural lookup (children and statuses render deterministically from frontmatter, via a small status-to-phrase mapping) and only the narrative halves consume synthesis tokens, so the full epic-aware brief fits the existing prompt envelope. The child aside is a one-time write at spawn time; drift on parent edits is an accepted v1 limitation. The promotion heuristic is a simple all-of-three gate, not a scoring function, and the suggestion is one line with decline as the default.

## Alternatives

- **Full v1 epic shape with epic-aware briefs** *(chosen)* — everything above. The pattern fit is clean: it extends two established brain shapes (skills as routers over underlying operations; computed reverse edges from authored frontmatter) without introducing a parallel pattern. The cost is v1 surface area — many small additions across the validator, `/shape`, `/zoom-out`, `/continue`, and governance docs; the fallback if any piece overshoots is the next option.
- **Same structure, frontmatter-only brief awareness** — identical except `/zoom-out` renders only the parent's slug and title on child briefs, with no umbrella narrative and no epic-targeted brief shape. Meaningfully smaller scope, but a reader landing on a child's brief sees that the parent exists without learning what the umbrella is about, and the navigation friction defeats the point for non-technical readers. Rejected because the audit showed the full shape fits the appetite — there was no forced trade-off.
- **Minimum viable: artifact only, defer all the help** — the kind, the folder, and the frontmatter field, with no pre-flight detection, no promotion suggestion, no brief awareness, no asides, and minimal validation. Ships fastest, but delivers the artifact shape with none of the brain's active help in maintaining or surfacing it — the same gap the PRD named. Rejected.
- **Do nothing** — rejected by the named user need and by the earlier [zoom-out decision](zoom-out-on-current-work.md), which had already chosen epic-awareness in briefs; inaction would force walking that choice back.

## Consequences

- **Closes** the umbrella gap: multi-PRD initiatives get a first-class artifact with active maintenance support across `/shape`, `/zoom-out`, and `/continue`, and the umbrella narrative is one inline paragraph away inside any child's brief.
- **Closes** the non-technical-reader navigation gap: the auto-prepended parent-epic aside gives a one-click path from any child back to the umbrella.
- **Opens** the mid-flight promotion question (a PRD that outgrows its appetite has no `--promote` flow in v1), the auto-closing question (v1 closes manually; auto-detecting "all children shipped" is deferred), the cross-scope-epic question (an umbrella spanning multiple repos is explicitly out of scope for v1), and the aside-drift question (spawn-time snapshots don't track parent edits).
- **Costs:** small schema and validator extensions with targeted test cases; protocol additions to three skill files plus one new template; and a grown discipline burden on the agent at child-spawn time (ask the detection question, set the frontmatter, append to the Children section, prepend the aside). If that discipline degrades as epics scale, auto-generating the Children section is the prioritised follow-up. The promotion heuristic also adds one small judgment step to every regular `/shape` pre-flight, mitigated by using only the pitch text already in context.

## Build notes

The build shipped with no deviations from the decision. The validator rules are vacuous-but-correct against a corpus with no epics yet; the `--epic` mode landed as its own phase section with the skipped phases named explicitly; the pre-flight detection and promotion steps landed adjacent, both defaulting to decline; the zoom-out target-kind branch landed with three explicit cases (epic target, child of an epic, everything else unchanged); and `/continue` gained state-table rows for epic targets and child lifecycle bumps. Every extension reused an established pattern — no parallel patterns, no new infrastructure. The quality of the epic-targeted brief lives in its synthesis prompt template inside the zoom-out skill; non-trivial future changes to it warrant a follow-up ADR.
